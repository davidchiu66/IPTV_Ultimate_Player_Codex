# IPTV Ultimate Player 需求与实施方案

## 1. 背景与目标

当前项目已经接入新版 `libmpv` 动态库，并且参考验证项目 `f:\Others\PYTHON\Mpv_Playe_CC\` 已证明新版 `libmpv` 可以支持部分 `MPD/DASH` 播放场景，包括带 `CENC` 加密且提供 `Keys` 的频道。

本次工作的目标不是单纯“升级 DLL”，而是围绕新版 `libmpv` 完成当前播放器的播放链路重构与能力补齐，使项目具备以下能力：

1. 支持 `cfg/json/m3u/m3u8/txt` 等不同格式来源的混合协议频道播放。
2. 支持 `MPD/DASH`、`HLS/M3U8`、以及其他常见流媒体协议的统一内置播放尝试。
3. 支持 `CENC` 加密频道的 `Keys` 注入播放。
4. 支持系统代理、用户代理、频道内代理的统一优先级管理。
5. 优先读取频道文件中的静态轨道信息，并在缺失时从实际播放流中动态补齐。
6. 在内置播放器失败时，统一进入浏览器播放兜底流程。

## 2. 设计原则

本次改造遵循以下核心原则：

1. 文件格式只决定“如何解析”，不决定“如何播放”。
2. 播放协议必须按单个频道、单条流独立判断，不能按文件扩展名判断。
3. 所有频道统一先尝试 `libmpv` 内置播放，失败后再进入浏览器兜底。
4. 静态频道元数据优先，运行时流媒体探测兜底。
5. 各类代理、请求头、DRM、轨道信息统一归一化后再进入播放层。
6. 变更范围优先收敛在现有架构内，避免无必要的大规模重写。

## 3. 当前问题概述

结合当前项目代码与参考项目分析，现阶段主要问题如下：

1. 当前项目在业务层主动拦截了 `MPD/DASH` 播放，以及部分 `DRM` 类型频道，导致新版 `libmpv` 的能力没有真正接入使用。
2. 当前播放逻辑主要围绕基础 `Manifest + UserAgent + Referer + Keys` 展开，对结构化 `cfg/json` 中更完整的元数据支持不足。
3. 当前解析逻辑更偏向“按文件类型粗分类”，而不是“按频道级别统一归一化”。
4. 代理能力缺少统一入口，尚未形成系统代理、用户代理、频道内代理的完整优先级链。
5. 轨道列表主要依赖 `mpv track-list` 动态获取，没有优先利用文件中已有的 `VideoList / AudioList / SubtitlesList`。
6. 对语言、分辨率、码率、编码等轨道信息的展示与优先级还不完整。

## 4. 范围定义

本次改造范围包括：

1. `backend/playlist_manager.py` 的解析增强与频道归一化。
2. `ui/player_panel.py` 的协议判断、DRM 参数注入、代理与请求头注入逻辑。
3. `ui/main_window.py` 的播放失败兜底交互。
4. 新增或补充代理配置持久化与读取逻辑。
5. 顶部轨道选择栏的数据来源增强与展示优化。

本次不作为首阶段目标的内容：

1. 直接重写为参考项目中的 `ctypes libmpv` 播放封装。
2. 大范围重构现有界面布局与整体交互风格。
3. 广泛修复历史文档或历史源码中的乱码文本。

## 5. 功能需求清单

### 5.1 新版 libmpv 播放适配

1. 使用当前项目中的新版 `plugins/Mpv/libmpv-2.dll` 作为内置播放器底层。
2. 移除当前业务层对 `MPD/DASH` 的预拒绝逻辑。
3. 不再根据 `DrmType` 或 `CdmType` 在播放前直接阻断频道。

### 5.2 混合协议频道支持

无论来源文件是 `cfg`、`json`、`m3u`、`m3u8`、`txt` 还是其他文本格式，只要其中包含多个协议混合的频道，都必须支持：

1. `MPD/DASH`
2. `HLS/M3U8`
3. `RTSP`
4. `RTMP`
5. 普通媒体直链
6. 其他 `libmpv` 可直接识别的地址

### 5.3 协议识别规则

每个频道的协议判断必须按以下优先级执行：

1. `ManifestType`
2. `ManifestInfo2.Type`
3. 流地址特征：
   - 包含 `.mpd` 或 `manifest?` 或 `dash` 时，识别为 `DASH`
   - 包含 `.m3u8` 或 `hls` 时，识别为 `HLS`
   - `rtsp://` 识别为 `RTSP`
   - `rtmp://` 识别为 `RTMP`
4. 无法识别时，标记为 `unknown` 并交给 `libmpv` 直接尝试

### 5.4 MPD + CENC 支持

对于 `MPD/DASH` 频道：

1. 若提供 `Keys`，需要从 `KID:KEY` 中提取纯 `KEY` 列表。
2. 在播放前向 `libmpv` 注入 `demuxer-lavf-o=cenc_decryption_key=...`。
3. 若频道无 `Keys`，仍应先尝试内置播放。
4. 播放失败后再进入浏览器兜底，不做播放前阻断。

### 5.5 浏览器兜底策略

对于所有频道，包括真正依赖浏览器 `EME` 的 `Widevine/PlayReady` 频道，统一按以下规则处理：

1. 先尝试 `libmpv` 内置播放。
2. 如果播放失败，弹出兜底选择框。
3. 选择框提供以下选项：
   - `使用默认浏览器播放`
   - `选择浏览器播放`
   - `放弃播放`

该行为属于固定运行时处理流程，不再视为阻断性风险。

### 5.6 代理优先级管理

代理使用优先级定义为：

1. 系统代理
2. 用户配置代理
3. 频道或文件中定义的代理字段补充
4. 无代理则直连

需要统一支持以下代理来源：

1. 系统代理
2. 用户手动配置代理
3. `Proxy`
4. `ManifestProxy`
5. `MediaProxy`
6. `ScriptProxy`

该代理策略需要覆盖：

1. `libmpv` 内置播放
2. 浏览器播放
3. 本地 HTTP 代理转发链路

### 5.7 用户代理设置与持久化

1. 需要提供用户代理配置的持久化文件。
2. 启动后默认优先读取系统代理。
3. 若系统无代理，则读取用户代理配置。
4. 若两者都无，则走直连。
5. 需要提供可供后续 UI 接入的配置读写能力。

### 5.8 cfg/json 解析增强

参考 `Mpv_Playe_CC`，当前项目需要增强对结构化频道文件的兼容解析，支持以下内容：

1. 频道集合入口：
   - `Channels`
   - `Streams`
   - `Items`
   - `Entries`
2. 流地址字段：
   - `Manifest`
   - `Url`
   - `URL`
   - `StreamUrl`
   - `StreamingUrl`
   - `PlaylistUrl`
   - `Mpd`
   - `MPD`
   - `Hls`
   - `HlsUrl`
3. 嵌套地址来源：
   - `ManifestInfo2`
   - `StreamingResolutions`
4. 代理字段：
   - `Proxy`
   - `ManifestProxy`
   - `MediaProxy`
   - `ScriptProxy`
5. 请求头字段：
   - `Headers`
   - `Headers2`
6. DRM 相关字段：
   - `Keys`
   - `License.Url`
   - `LicenseUrl`
   - `Drm`
   - `DrmType`
   - `CdmType`

### 5.9 m3u/txt 等文本格式混合协议支持

对于 `m3u/m3u8/txt` 以及其他文本型文件：

1. 解析器仅负责抽取频道条目与附带元数据。
2. 不能因为文件是 `m3u` 就假设所有频道都是 `HLS`。
3. 每条频道在进入播放层之前，都必须执行统一协议判断与字段归一化。

### 5.10 轨道静态信息优先解析

优先从频道文件中读取以下轨道相关字段：

1. `VideoList`
2. `AudioList`
3. `SubtitlesList`
4. `DefaultVideo`
5. `DefaultAudio`
6. `DefaultSubtitles`

### 5.11 轨道动态探测兜底

若文件中缺少轨道信息，则在播放成功后通过 `libmpv` 获取运行时轨道信息，包括但不限于：

1. `track-list`
2. `video_params`
3. `duration`
4. 其他可用的媒体元数据

### 5.12 轨道展示增强

轨道展示应尽量支持以下探测与展示：

1. 语言
2. 分辨率
3. 码率
4. 编码格式
5. 帧率
6. 声道数
7. 采样率

轨道信息优先级规则为：

1. 文件内静态轨道元数据优先。
2. 运行时探测结果用于补齐缺失字段，必要时修正明显不准确的静态字段。

## 6. 内部统一数据模型

为避免解析层与播放层各自判断，所有来源文件在进入播放层前，均应归一化为统一频道结构。建议统一包含以下字段：

1. `Name`
2. `Category`
3. `Manifest`
4. `ManifestType`
5. `DrmType`
6. `CdmType`
7. `LicenseUrl`
8. `Keys`
9. `Headers`
10. `UserAgent`
11. `Referer`
12. `Proxy`
13. `VideoTracks`
14. `AudioTracks`
15. `SubtitleTracks`
16. `DefaultVideo`
17. `DefaultAudio`
18. `DefaultSubtitles`

## 7. 实施方案

### 第一阶段：解析层归一化改造

目标：让所有来源文件统一输出频道对象。

实施内容：

1. 升级 `backend/playlist_manager.py`。
2. 增加统一的频道归一化方法。
3. 将 `cfg/json/m3u/m3u8/txt` 解析结果统一映射到内部频道模型。
4. 补充对 `ManifestType`、`ManifestInfo2`、`StreamingResolutions`、`Keys`、`Headers`、`Proxy` 系列字段的提取。
5. 增加轨道静态信息提取逻辑。

预期结果：

1. 上层播放模块不再依赖原始文件格式差异。
2. 各种来源文件都能得到结构一致的频道对象。

### 第二阶段：代理与配置层建设

目标：统一代理获取与持久化逻辑。

实施内容：

1. 新增配置文件，例如 `config/app_settings.json`。
2. 新增代理读取与合并模块。
3. 支持：
   - 读取系统代理
   - 读取用户配置代理
   - 合并频道级代理字段
4. 为后续 UI 接入提供配置读写接口。

预期结果：

1. 内置播放与浏览器播放使用一致的代理决策结果。
2. 代理来源清晰可控。

### 第三阶段：播放器播放链路改造

目标：让 `libmpv` 真正成为所有频道的统一首选播放入口。

实施内容：

1. 修改 `ui/player_panel.py`。
2. 移除现有 `MPD/DASH` 和 `DRM` 的播放前阻断逻辑。
3. 增加频道级协议识别。
4. 增加 `MPD + Keys` 的 `cenc_decryption_key` 注入。
5. 增加代理、`UserAgent`、`Referer`、`Headers` 注入。
6. 切台时清理残留的：
   - `demuxer-lavf-o`
   - `http-proxy`
   - `http-header-fields`
   - `user-agent`
   - `referrer`

预期结果：

1. 所有频道统一先走内置播放。
2. 混合协议文件中的不同频道能分别按协议播放。

### 第四阶段：轨道优先级与展示改造

目标：在轨道选择与展示上优先使用文件元数据，并允许运行时补齐。

实施内容：

1. 将文件中的 `VideoList / AudioList / SubtitlesList` 映射为内部轨道结构。
2. 播放前优先将静态轨道信息提供给顶部轨道栏。
3. 播放后如 `mpv track-list` 提供更完整信息，则补齐语言、分辨率、码率、编码等字段。
4. 优化 `ui/player_controls.py` 中轨道展示文案。

预期结果：

1. 用户在播放前后都能看到更丰富、稳定的轨道信息。
2. 对 `tennischannels.cfg` 这类结构化源的利用率显著提升。

### 第五阶段：播放失败兜底交互

目标：将失败场景明确纳入既定播放流程。

实施内容：

1. 修改 `ui/main_window.py` 的播放失败处理逻辑。
2. 将当前单一警告框改为三选项兜底对话框。
3. 接入现有：
   - 默认浏览器播放逻辑
   - 选择浏览器播放逻辑
   - 放弃播放逻辑

预期结果：

1. `Widevine/PlayReady` 等无法内置播放的频道自动进入可选兜底。
2. 用户始终有明确下一步操作选择。

## 8. 风险与边界

当前已确认的主要风险点如下：

1. 某些频道的 `Headers`、`ManifestProxy`、`MediaProxy` 存在作用域差异，首版如果统一注入，个别源可能仍需二次细化。
2. `python-mpv` 与新版 `libmpv` 在部分 Windows 环境下可能存在边缘兼容性问题；若后续发现稳定性不足，再评估迁移参考项目中的 `ctypes libmpv` 封装方案。
3. 文件中的静态轨道信息与实际流返回的 `track-list` 可能不完全一致，需要定义明确的合并策略，避免界面与实际选择状态不一致。

以下事项不再视为阻断性风险，而是已定义好的运行时兜底策略：

1. 真正依赖浏览器 `EME` 的 `Widevine/PlayReady` 频道，先尝试 `libmpv`，失败后弹出浏览器兜底选择框。

## 9. 验证方案

实施完成后，至少应覆盖以下验证场景：

1. `tennischannels.cfg`
   - 验证同一文件中 `MPD` 与 `M3U8` 混合协议可正常识别与播放
   - 验证静态轨道列表读取正常
2. 带 `cenc.mpd + Keys` 的频道
   - 验证 `cenc_decryption_key` 注入有效
3. 普通 `m3u8` 频道
   - 验证 `HLS` 走内置播放
4. 无静态轨道信息的频道
   - 验证运行时 `track-list` 能补齐轨道展示
5. 系统代理场景
   - 验证系统代理优先生效
6. 用户代理场景
   - 验证系统无代理时使用用户代理
7. 直连场景
   - 验证两者都无时走直连
8. 内置播放失败场景
   - 验证三选项兜底弹窗逻辑正确

## 10. 实施顺序建议

建议按以下顺序推进：

1. 先完成解析层与统一数据模型改造。
2. 再完成代理与配置层建设。
3. 然后改造 `libmpv` 播放链路。
4. 再补轨道展示与元数据优先级。
5. 最后接入播放失败兜底交互。

这样可以先确保“数据拿对”，再确保“播放动作正确”，最后再补足体验层。

## 11. 结论

本次改造的本质不是单一修补 `MPD` 支持，而是将当前项目从“按文件格式和少量字段做播放判断”的模式，升级为“按频道级统一归一化 + 按协议级播放决策 + 按轨道级元数据管理”的完整播放架构。

在该方案下，`MPD/DASH`、`HLS/M3U8`、混合协议文件、`CENC` 加密流、代理策略、轨道元数据与浏览器兜底，将统一纳入同一条播放链路管理，形成可维护、可扩展的新版 `libmpv` 适配方案。
