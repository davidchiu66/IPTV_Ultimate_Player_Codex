# 项目规划与提示语协作复盘总结

更新时间：2026-06-24

## 一、阶段性总体结论

IPTV Ultimate Player Codex 已经从最初的 IPTV/本地媒体播放器，推进成了一个具备较完整产品雏形的 Windows 桌面播放器。当前项目已经覆盖播放核心、资源管理、频道管理、收藏、播放列表、在线资源、浏览器辅助解析、诊断日志、UI 统一、GitHub 发布、CI 打包、安装包、宣传网页、兼容模式等多个方面。

从项目阶段看，当前已经不再是单纯的“功能能否实现”阶段，而是进入了“产品化稳定、架构收束、体验一致、回归治理”的阶段。下一阶段的核心目标应当从快速堆功能，转向降低返工、稳定质量、沉淀规范、建立测试样本库。

## 二、已经形成的主要成果

1. 播放核心

- 基于 PySide6 QWidget + libmpv 原生 DLL 集成。
- 支持 HLS、DASH/MPD、MP4、FLV、本地视频、本地音频、图片、GIF 等资源。
- 支持播放/暂停、停止、进度、音量、倍速、全屏、上一项/下一项。
- 增加 Windows 10 兼容/安全启动模式，用于规避 Qt6 图形后端、D3D11、libmpv、无边框窗口组合导致的黑屏风险。

2. 资源与频道体系

- 支持资源目录扫描。
- 支持频道资源文件、视频、音频、GIF、图片等资源类型。
- 支持频道列表、频道分组、频道详情、频道收藏、资源收藏。
- 支持资源收藏和频道收藏的查看、播放、删除关联关系、失效检测、搜索和筛选。

3. 在线资源能力

- “打开在线 m3u”升级为“打开在线资源”。
- 支持在线频道文件：m3u、txt、cfg、json。
- 支持直接打开在线视频流、音频流、图片、GIF、HLS、DASH。
- 下载、探测、播放链路支持 HTTPS，并可跳过证书验证。
- 增加在线资源探测/下载进度提示、失败提示、0 频道解析提示。
- 在线资源进度弹窗已统一到玻璃拟态风格。

4. 播放列表能力

- 支持本地播放专辑。
- 支持从文件夹创建专辑。
- 支持专辑持久化。
- 支持自动连播、跳过片头、跳过片尾等专辑级设置。

5. UI 与体验

- 播放控制栏、工具栏、顶部条、资源面板、播放列表、设置面板逐步统一为玻璃拟态风格。
- 优化菜单、全屏、音量、播放按钮、倍率、进度条等视觉细节。
- 全屏状态下支持鼠标空闲自动隐藏。
- 加载直播流时优化了加载提示和侧边面板协同。

6. 工程化

- 增加 README、英文 README、ABOUT、LICENSE。
- 增加 GitHub Actions Windows 打包工作流。
- 完成 GitHub 仓库推送和发布准备。
- 增加宣传网页。
- 更新 requirements.txt，明确 Python 依赖与 libmpv DLL 的关系。

## 三、规划设计方面存在的问题和不足

### 1. 项目范围扩张过快

项目推进过程中，UI、播放解析、资源收藏、频道收藏、播放列表、浏览器播放、GitHub 发布、打包、宣传页、兼容模式等任务交织推进。短期看推进速度很快，但中后期容易带来三个问题：

- 一个小需求牵动多个模块。
- 修改边界不清晰，容易出现连锁回归。
- 用户体验问题和底层技术问题混在一个开发节奏里，排查成本上升。

改进建议：下一阶段将需求拆成三条主线分别推进：稳定性主线、架构主线、产品体验主线。每次只推进一条主线中的明确任务，减少跨层级混杂。

### 2. UI 目标缺少稳定设计基准

播放控制栏和整体界面经历了多轮基于原型图、截图、SVG、描述词的调整。虽然最终视觉明显改善，但过程中出现了反复调整字号、间距、光晕、按钮尺寸、标题栏色调等情况。

根本原因是早期没有形成稳定的设计规格表。比如：

- 控件尺寸没有统一 token。
- 颜色、透明度、边框、阴影没有统一变量表。
- 弹窗、面板、标题栏、按钮的风格规范是后期逐步补齐的。

改进建议：下一阶段建立《UI 设计规范.md》，至少包含：颜色 token、圆角、边框、阴影、字体、按钮尺寸、面板间距、弹窗样式、加载状态、空状态、错误状态。

### 3. 功能语义多次后置澄清

项目中多次出现需求语义后续扩展或修正的情况，例如：

- “播放控制面板不要中文”后来澄清为只限播放控制面板，不是所有面板。
- “打开在线 m3u”后来升级为“打开在线资源”。
- 左边缘滑入规则从频道列表，扩展为根据直播、本地资源、频道收藏、频道列表是否为空等状态决定。
- 上一个/下一个按钮从简单切换，扩展为频道收藏、频道列表、本地资源、资源收藏、播放专辑等不同上下文。

这些变化本身合理，但说明早期领域模型没有完全定型。

改进建议：对关键功能先写状态机或上下文规则，再编码。例如“当前播放来源”应提前抽象为统一枚举：channel_list、channel_favorites、resource_files、resource_favorites、playlist_album、online_resource。

### 4. 验收标准一度偏主观

部分 UI 反馈使用了“差强人意”“不协调”“不好看”“太丑”“接近目标”等描述。这类反馈对产品判断很真实，但对实现不够可测，容易导致模型猜测式修改。

改进建议：将主观反馈拆成可执行指标，例如：

- 字号从 12px 调整到 16px。
- 控制栏高度为 120px。
- 按钮间距为 14px。
- 进度条高度为 4px。
- 鼠标无活动 2.5 秒隐藏。
- 在线资源解析出 0 频道时必须弹窗提示。

### 5. 缺少固定回归测试样本库

项目中已经出现过大量代表性样本：

- GDTV 嗅探型直播。
- KBSworld 403/Referer/User-Agent 问题。
- XHS 回放 MP4 重定向。
- GitHub raw 在线 m3u。
- MPD/CENC。
- HLS 浏览器失败但 libmpv 可播。
- Windows 10 全屏黑屏。

这些样本非常宝贵，但目前主要散落在对话和日志中。

改进建议：建立 `docs/test-matrix.md` 或 `tests/manual_cases.md`，记录每个样本的 URL、资源类型、预期表现、历史问题、当前状态、回归步骤。

### 6. 架构治理滞后于功能推进

很多架构性能力是在问题暴露后补上的：

- 日志体系是在崩溃排查后加强。
- 兼容模式是在 Windows 10 黑屏后加入。
- requirements 是 PySide6 缺失后检查补充。
- UI 风格统一是在多个旧界面存在色差后集中整改。

这符合快速迭代阶段，但下一阶段需要从被动修补转为主动治理。

改进建议：每进入一个新阶段，先做一次架构盘点，明确哪些模块需要先抽象、哪些技术债必须先还。

## 四、提示语使用方面存在的问题

### 1. 信息很完整，但结构有时混杂

很多提示同时包含目标、约束、实现方式、验收标准和审美判断。信息量足够，但模型需要二次拆解，容易遗漏优先级。

改进建议：使用固定需求模板：

```text
目标：
背景：
范围：
不做：
交互规则：
数据/状态：
异常处理：
验收方法：
优先级：
参考文件/截图：
```

### 2. 强否定较多，正向规格有时不足

例如“不要 Bootstrap 风格”“不要普通桌面软件风格”“不要自行发挥”等提示能避免方向跑偏，但如果没有正向规格，模型仍需要猜测目标风格。

改进建议：否定约束后面补正向规格。例如：

```text
不要 Bootstrap 风格。
目标风格：Apple TV / macOS glassmorphism。
颜色：#1A2333 到 #3A4456 深灰蓝金属渐变。
边框：rgba(120,180,255,100)。
圆角：20px。
阴影：蓝色柔光，不要大面积亮蓝网页感。
```

### 3. “重试”类指令容易导致猜测

单独说“重试”时，如果没有指出上次失败在哪里，模型只能根据上下文猜测，容易在同一问题附近反复。

改进建议：重试时使用三行格式：

```text
重试。
上次问题：按钮组仍偏右，音量条没有垂直居中。
这次只改：按钮组整体左移 20px，音量条下移 4px。
```

### 4. 截图和原型引用需要版本锚点

长线程里“图1”“图2”“上图”“原型图”容易漂移。

改进建议：给参考图固定命名并说明优先级：

```text
视觉基准：播放器播放控制条UI原型设计图2.png。
运行截图：screenshot_20260624_current.png。
以视觉基准为准，运行截图只用于指出当前差异。
```

### 5. 不同层级需求应拆开

UI 微调、播放解析、打包发布、兼容性问题属于不同层级。放在一个任务中会增加上下文污染和回归风险。

改进建议：按任务类型拆分：

- UI 任务：只改样式、布局、动效。
- 播放任务：只改解析、播放、请求头、mpv 参数。
- 工程任务：只改打包、依赖、CI、安装脚本。
- 文档任务：只改 README、方案、总结。

## 五、推荐的下一阶段协作流程

```

这个流程之前已经多次使用，效果明显更好。下一阶段建议固定下来，除紧急 bug 外不要跳过。

git status --short
```

## 六、下一阶段建议优先级

### P0：稳定性与回归样本库

- 建立手工测试矩阵。
- 固化典型直播源、回放源、本地媒体、在线资源样本。
- 每次播放链路修改后按清单回归。

### P1：数据模型收束

重点梳理：

- 频道资源文件。
- 单个频道。
- 本地媒体资源。
- 在线媒体资源。
- 收藏关系。
- 播放队列上下文。
- 播放专辑。

目标是减少 `if channel_list / resource_files / favorites / playlist_album / online_resource` 这类分散判断。

### P2：UI 规范化

建立统一 UI token 和组件规范：

- 面板。
- 弹窗。
- 标题栏。
- 按钮。
- 输入框。
- 列表。
- 加载状态。
- 空状态。
- 错误状态。

### P3：发布与兼容性治理

- Windows 10/11 兼容矩阵。
- 安全模式说明。
- 依赖检测。
- libmpv 缺失诊断。
- 安装包目录结构检查。
- GitHub Release 自动发布链路完善。

## 七、结论

当前项目推进速度很快，功能覆盖面已经比较广。主要短板不是执行能力，而是需求边界、设计基准、测试样本和架构抽象需要进一步前置。

下一阶段最重要的变化应当是：

- 少做模糊试探，多做规格化输入。
- 少做单点修补，多做状态机和模型收束。
- 少靠截图感觉反复调，多建立 UI token 和验收清单。
- 少靠问题发生后补救，多建立回归样本和兼容矩阵。

如果后续能坚持“先方案、再落档、再编码、再验收”的节奏，项目质量和协作效率都会明显提升。
