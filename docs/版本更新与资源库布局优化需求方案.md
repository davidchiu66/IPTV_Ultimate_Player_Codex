# 版本更新与资源库布局优化需求方案

## 1. 背景

当前 `关于` 界面已经提供 `检查版本更新` 和 `在线更新` 按钮，但实际行为只是打开 GitHub Releases 页面，没有完成应用内版本检测、下载进度、安装版/便携版更新路径区分。

资源库面板当前按钮纵向排列，占用较多高度，资源文件列表可视区域偏小。需要把按钮重排为更紧凑的两列布局，并补充“打开资源文件”入口，让用户可以直接打开单个频道文件、视频、音频、图片或 GIF。

## 2. 需求分析

### 2.1 检查版本更新

目标：

1. 从 GitHub 获取当前项目最新 Release 版本。
2. 与本地版本 `app_version.txt` / `APP_VERSION` 比较。
3. 若远端版本高于本地版本，提示用户有新版本。
4. 若远端版本不高于本地版本，提示当前已是最新版本。
5. 用户只检查版本时，不强制更新。

当前版本来源：

- 本地：`utils.app_version.get_app_version()`。
- 远端：GitHub Releases API，建议优先调用：

```text
https://api.github.com/repos/davidchiu66/IPTV_Ultimate_Player_Codex/releases/latest
```

版本比较规则：

1. 去掉远端 tag 中的 `v` / `V` 前缀。
2. 支持 `0.1.9`、`0.1.9-dev`、`v0.1.9`。
3. 比较主版本号数字段，`0.1.10` 应大于 `0.1.9`。
4. `dev`、`alpha`、`beta`、`rc` 等预发布标记低于正式版。
5. 若远端版本无法解析，则给出“无法识别远端版本”的提示，不做强制更新。

### 2.2 在线更新

目标：

1. 在线更新前先检查 GitHub 是否有高于本地版本的新版本。
2. 若没有新版本，提示用户当前已是最新版本。
3. 若有新版本，提示版本号、Release 页面、更新说明摘要，并由用户确认是否下载。
4. 用户确认后下载对应资产，显示下载进度条。
5. 下载完成后启动更新流程。
6. 一旦启动安装或覆盖更新，应明确提示“程序将强制退出当前已打开的播放器”。

安装版更新策略：

1. 从 Release assets 中选择 Windows 安装包：

```text
IPTVPlayer-Setup-<version>.exe
```

2. 下载到 `%LocalAppData%\IPTV_Ultimate_Player_Codex\updates\`。
3. 下载完成后提示用户即将退出当前程序并启动安装器。
4. 启动安装器后退出当前应用。
5. 安装器负责覆盖安装。

便携版更新策略：

1. 从 Release assets 中选择 Windows 便携包：

```text
IPTV_Ultimate_Player_Codex-Portable-<version>.zip
```

或现有工作流生成的：

```text
<APP_NAME>-Portable-<version>.zip
```

2. 下载到 `%LocalAppData%\IPTV_Ultimate_Player_Codex\updates\`。
3. 当前进程不能直接覆盖自身运行目录，需要生成外部 updater 脚本。
4. Updater 脚本职责：
   - 等待当前播放器进程退出；
   - 解压 zip 到临时目录；
   - 覆盖便携版程序目录；
   - 可选启动新版本主程序；
   - 写入更新日志。
5. 启动 updater 脚本后退出当前应用。

运行形态判断：

1. 若 `sys.frozen` 为 False，视为源码开发模式。
2. 源码开发模式不做自动覆盖更新，只提示并打开 Release 页面。
3. 若为 PyInstaller 打包：
   - 如果可检测到安装目录在 `Program Files` 或有卸载信息，可优先视为安装版。
   - 若可执行文件目录旁存在 `README_PORTABLE.txt`，视为便携版。
   - 无法可靠判断时，提示用户选择“下载安装包”或“下载便携包”，默认推荐安装包。

异常处理：

1. GitHub API 请求失败：提示网络错误，可引导用户检查代理或打开 Release 页面。
2. GitHub 限流：提示稍后重试。
3. 没有找到匹配资产：提示“Release 中未找到适合当前系统的安装包/便携包”。
4. 下载中断：保留临时文件或删除 `.part` 文件，提示可重试。
5. 启动安装器或 updater 失败：弹窗提示错误，并保留已下载文件路径。

### 2.3 资源库面板按钮重排

目标：

1. 缩短当前所有按钮长度，按钮保持一致高度和宽度。
2. 第一排：`打开资源目录` + `打开在线资源`。
3. 第二排：`打开资源文件` + `刷新当前资源`。
4. `刷新当前目录` 文案调整为 `刷新当前资源`，语义覆盖当前资源文件或目录。
5. 第三排：`删除选中文件` + `取消`。
6. 为资源文件列表留出更多纵向空间。

新增“打开资源文件”能力：

1. 使用 `QFileDialog.getOpenFileName()`。
2. 可选择播放器当前支持的单个资源格式：
   - 频道文件：`m3u`、`m3u8`、`txt`、`cfg`、`json` 等；
   - 视频：现有 `LOCAL_VIDEO_EXTENSIONS`；
   - 音频：现有 `LOCAL_AUDIO_EXTENSIONS`；
   - 图片：现有 `LOCAL_IMAGE_EXTENSIONS`；
   - GIF：现有 `LOCAL_GIF_EXTENSIONS`。
3. 选择频道文件后复用 `_on_file_selected()` 加载频道列表。
4. 选择本地媒体后复用 `_play_local_media_file()` 直接播放。
5. 选择文件所在目录可同步为当前资源上下文，后续刷新时刷新该文件所在目录。

刷新语义调整：

1. 当前上下文是目录：刷新目录。
2. 当前上下文是单个资源文件：
   - 若文件仍存在，则刷新其所在目录，并在列表中保留/选中该文件；
   - 若文件不存在，提示文件已失效。
3. 如果没有上下文，刷新默认用户资源目录。

## 3. 技术方案

### 3.1 新增版本更新服务

建议新增模块：

```text
utils/update_manager.py
```

核心职责：

1. 读取本地版本。
2. 请求 GitHub latest release。
3. 解析版本号、Release HTML URL、assets、body。
4. 比较版本。
5. 按运行形态选择安装包或便携包资产。
6. 下载资产并回调下载进度。

建议数据结构：

```python
@dataclass
class ReleaseInfo:
    version: str
    tag_name: str
    html_url: str
    body: str
    assets: list[dict]

@dataclass
class UpdateCheckResult:
    local_version: str
    latest_version: str
    has_update: bool
    release: ReleaseInfo | None
    message: str
```

版本比较函数：

```python
compare_versions(local: str, remote: str) -> int
```

返回值：

- `-1`：local 低于 remote；
- `0`：相等；
- `1`：local 高于 remote；
- `None` 或抛出受控异常：无法比较。

### 3.2 AboutDialog 与 MainWindow 协同

当前 `AboutDialog` 内按钮直接打开 Release 页面，建议改成信号驱动：

```python
check_update_requested = Signal()
online_update_requested = Signal()
```

按钮点击后发信号，由 `MainWindow` 执行实际逻辑：

1. `check_update_requested` -> `MainWindow.check_for_updates()`
2. `online_update_requested` -> `MainWindow.start_online_update()`

这样可以复用主窗口已有的统一弹窗、进度弹窗和线程管理能力。

### 3.3 后台线程与进度弹窗

版本检查和下载不能阻塞 UI。

建议实现：

1. 使用 `QThread` + worker，或沿用项目中已有线程风格。
2. 检查版本时显示轻量状态提示或忙碌弹窗。
3. 下载更新包时显示统一玻璃拟态进度弹窗。

进度弹窗要求：

1. 标题：`正在下载更新`。
2. 文案显示版本号和资产名称。
3. 进度条显示百分比。
4. 显示已下载 / 总大小。
5. 支持取消下载。
6. 风格与当前 `message_dialogs` / 在线资源进度弹窗一致。

### 3.4 安装版启动更新

下载完成后：

1. 弹窗确认：

```text
安装包已下载完成。
启动更新后，当前播放器将强制退出。
是否立即开始安装？
```

2. 用户确认后：
   - `subprocess.Popen([installer_path])`
   - 调用主窗口关闭流程；
   - 必要时 `QApplication.quit()`。

注意：

1. 不在当前进程中等待安装器结束。
2. 不删除安装包，便于失败后用户手动运行。

### 3.5 便携版覆盖更新

新增 updater 脚本生成逻辑：

1. 在 `%LocalAppData%\IPTV_Ultimate_Player_Codex\updates\` 生成 `apply_update.ps1` 或 `.bat`。
2. 脚本参数：
   - 当前进程 PID；
   - zip 路径；
   - 应用目录；
   - exe 路径；
   - 临时解压目录。
3. 脚本流程：
   - 等待 PID 退出；
   - 解压 zip；
   - 将包内文件覆盖到应用目录；
   - 启动新 exe；
   - 写入 `update_apply.log`。

安全限制：

1. 应用目录必须是当前 exe 所在目录。
2. 解压结果必须包含主程序 exe。
3. 不允许覆盖根目录、系统目录或空路径。
4. 出错时保留日志并提示用户手动更新。

### 3.6 资源库布局实现点

1. 新增信号：

## 4. 风险与约束

1. GitHub API 访问可能受网络、代理、限流影响，需要明确错误提示。
2. GitHub Release 资产命名必须稳定，否则自动选择安装包/便携包会失败。
3. 当前程序不能覆盖正在运行的自身文件，便携版必须走外部 updater。
4. 安装版在 `Program Files` 下覆盖安装可能触发 UAC，安装器应负责权限提升。
5. 源码开发模式不应执行自动覆盖，避免误改工作区。
6. 更新过程中强制退出前，应提示用户可能中断当前播放。
7. 下载文件需要校验大小，后续可扩展 SHA256 校验。

## 5. 验收方法

### 5.1 检查版本更新

1. 本地版本低于 GitHub latest release：点击 `检查版本更新`，弹窗提示有新版本，显示本地版本和远端版本。
2. 本地版本等于或高于 latest release：提示当前已是最新版本。
3. 断网或 GitHub 不可访问：提示检查失败，并给出打开 Release 页面选项。
4. 远端版本无法解析：提示无法识别远端版本，不触发更新。

### 5.2 在线更新

1. 有新版本时点击 `在线更新`，先展示新版本确认弹窗。
2. 用户确认后出现下载进度条，进度随下载推进。
3. 用户取消下载后，下载停止并提示已取消。
4. 安装版下载完成后，确认启动安装器，当前程序退出。
5. 便携版下载完成后，确认启动 updater，当前程序退出，updater 完成解压覆盖。
6. 源码模式点击在线更新，不覆盖代码，只提示打开 Release 页面或手动下载。
7. Release 中没有匹配资产时，提示缺少安装包/便携包。

### 5.3 资源库按钮布局

1. 资源库目录 Tab 第一排显示 `打开资源目录` + `打开在线资源`，两个按钮同宽。
2. 第二排显示 `打开资源文件` + `刷新当前资源`，两个按钮同宽。
3. `删除选中文件` 与 `取消` 同排且同宽。
4. 文件列表可视高度比当前版本明显增加。
5. 点击 `打开资源文件` 可选择单个频道文件，并加载频道列表。
6. 点击 `打开资源文件` 可选择本地视频/音频/图片/GIF，并直接播放。
7. 刷新当前资源在目录和单文件上下文都能正确工作。
8. 资源文件列表仍保持长文件名省略、收藏星标可见、无横向滚动条。

## 6. 建议实施顺序

1. 先完成资源库按钮重排和“打开资源文件”，风险较低、用户体验收益直接。
2. 再实现版本检查，先只提示，不下载。
3. 再实现安装版在线更新下载和启动安装器。
4. 最后实现便携版 updater 覆盖更新。
