# IPTV 播放器阶段性开发总结与下一阶段计划

更新时间：2026-06-19

## 一、当前阶段开发总结

本阶段的核心目标是将项目从老版本 libmpv 迁移到新版 libmpv，并围绕 MPD/DASH、HLS、网页型直播源、播放失败兜底和界面交互进行连续改造。当前已经进入“主要播放链路可用，复杂源继续增强”的阶段。

### 1. 新版 libmpv 播放核心

- 已将新版 libmpv 作为主要内置播放核心。
- 普通 HLS/M3U8 频道已恢复正常播放。
- MPD/DASH 频道已能进入内置播放器播放。
- 带 Keys 的 MPD/CENC 频道已经取得实质性进展，能够起播并输出音视频。
- 已处理过新版 libmpv 初始化、黑屏、有声无画、串台、旧后台声音残留等问题。
- 播放状态的加载动画、加载文案、起播等待逻辑已完成第一轮优化。

### 2. 播放失败与浏览器兜底

- 播放策略已调整为：所有频道先尝试 libmpv 内置播放器播放。
- 内置播放失败后，根据失败类型弹窗提示用户选择：
  - 默认浏览器播放；
  - 选择浏览器播放；
  - 重试探测；
  - 取消播放；
  - 对明确失效的链接提示是否播放下一个。
- 对真正依赖浏览器 EME 的 Widevine/PlayReady 场景，保留浏览器兜底策略。

### 3. 协议识别与混合源处理

- 已明确不再按频道文件格式简单判断播放协议。
- `cfg`、`m3u`、`txt` 等文本来源中可以混合 MPD、M3U8、MP4、FLV、网页型直播地址等。
- 播放时按单个频道链接和元数据进行协议判断。
- `Channels\tennischannels.cfg` 这类 MPD/HLS 混合场景已纳入设计目标。

### 4. 网页型直播源探测

- 已完成 HLS 智能解析第一阶段。
- 对无明确后缀、实际为网页播放页的频道，增加了浏览器探测与静态/动态嗅探。
- 已支持从 GDTv 这类页面中探测真实直播流，并交给 libmpv 播放。
- 探测关键词维度已扩展，包括：
  - `m3u8`
  - `mpd`
  - `flv`
  - `ts`
  - `m4s`
  - `manifest`
  - `playlist`
  - `stream`
  - `play`
  - `live`
  - `channel`
  - `media`
  - `video`
- HTTP 状态码处理已从仅判断 `200 OK` 扩展到合理处理 `2xx`、常见 `3xx`、`206 Partial Content` 以及明确失败的 `4xx/5xx`。

### 5. 代理与配置

- 已支持系统代理优先。
- 若系统代理不存在，则读取用户配置代理。
- 若系统代理和用户代理都不存在，则直连。
- 探测超时时长已支持配置，默认已延长到更适合复杂页面的时长。
- 浏览器播放端口、用户代理等相关设置已逐步纳入配置体系。
- 已新增右侧滑出的设置界面，用于集中管理代理、探测超时、浏览器端口等参数。

### 6. 界面调整

- 顶部工具栏已简化：
  - 左侧为频道资源按钮；
  - 旁边为打开在线 m3u；
  - 中间为“IPTV 播放器”；
  - 右侧为设置按钮。
- 应用标题已从 `Python` 调整为 `IPTV 播放器`。
- 播放状态顶部条保留并整合了：
  - 频道文件列表；
  - 频道详情；
  - 停止播放；
  - 全屏 F1；
  - 加载节目单；
  - 下载节目单。
- 左侧滑出面板已调整为频道分组和频道列表。
- 停止播放后，播放界面和顶部条会消失，回到初始状态。
- 已修复切换频道文件后，点击频道播放时顶部条频道名称不及时刷新的问题。

## 二、当前遗留问题

### 1. iptv345 类复杂网页源

该类网站在网页上可以播放，但在播放器中仍存在失败场景。当前观察到的特点包括：

- 需要先触发页面内置播放按钮，才生成真实直播源。
- 每个频道可能存在多线路选择。
- 真实 m3u8 带动态 session 参数，例如：
  - `https://t2.iptv200.com/live/cctv1/index.m3u8?session=...`
- 探测到 m3u8 后，交给 libmpv 播放仍可能返回 `404`。
- 需要继续分析 session、Referer、Origin、User-Agent、Cookie、线路选择和时效性问题。

该问题已记录到遗留问题清单，暂不阻塞主线开发。

### 2. 复杂网页源成功率仍需提升

GDTv 类页面已经可以成功播放，但仍有进一步提升空间：

- DOM 定点抓流稳定性需要继续增强。
- 页面播放器对象、JS 变量、网络请求、媒体标签、Hls.js/mpegts 实例等多维度探测需要继续融合。
- 候选源评分和验证机制需要进一步完善，避免误选图片、脚本、广告或统计请求。

### 3. 轨道与媒体信息探测仍需补齐

后续仍需完善：

- 优先从频道文件中读取视频轨、音频轨、字幕轨。
- 文件没有轨道信息时，再从播放流的 meta 信息中获取。
- 补齐语言、分辨率、码率探测。
- 将轨道信息稳定显示到播放顶部条菜单中。

## 三、下一阶段开发计划建议

下一阶段建议先做稳定性收口，再继续扩大复杂网页源覆盖率。

### 第一阶段：播放切换稳定性回归

目标：确保主播放链路稳定，不出现旧问题回退。

重点验证：

- 切换频道文件后播放新频道；
- 快速切换频道；
- HLS 与 MPD 混播；
- 普通 HLS 播放；
- 普通 MPD 播放；
- MPD/CENC 播放；
- 停止播放后回到初始状态；
- 播放失败后的重试、浏览器兜底、取消播放；
- 后台不残留旧频道声音；
- 代理请求不残留旧频道流量；
- 顶部条标题、频道详情、轨道菜单与当前频道一致。

### 第二阶段：媒体解析器结构整理

目标：让所有播放入口都经过统一、可扩展的解析流程，但避免一次性解析整个频道文件导致耗时。

建议结构：

- 单频道按需解析；
- 候选源收集；
- 候选源去重；
- 候选源评分；
- 候选源轻量验证；
- 输出最终可播放地址；
- 失败时返回明确原因。

候选源评分应优先考虑：

- 明确的 `.m3u8`、`.mpd`、`.flv`、`.mp4`；
- `Content-Type` 为 HLS/DASH/视频类型；
- Chrome Network 中真实媒体请求；
- Hls.js/mpegts/video.js 等播放器实例中的当前源；
- 页面 DOM 中播放器相关字段；
- 避免误选 `.jpg`、`.jpeg`、`.png`、`.gif`、`.css`、统计脚本、广告资源。

### 第三阶段：复杂网页源专项增强

目标：继续提高 GDTv、iptv345 等网页型频道成功率。

建议方向：

- 增强 GDTv DOM 定点抓流。
- 对需要用户动作的网站，模拟点击播放按钮。
- 支持线路按钮识别与线路切换探测。
- 捕获动态生成的 session m3u8。
- 播放前保留必要请求头：
  - Referer；
  - Origin；
  - User-Agent；
  - Cookie；
  - Accept；
  - Range。
- 对带时效 session 的链接，缩短“探测到源”和“交给 libmpv 播放”之间的间隔。

### 第四阶段：轨道与媒体信息完善

目标：让播放顶部条的轨道信息更准确、更完整。

计划内容：

- 解析频道文件中的视频轨、音频轨、字幕轨配置。
- 从流 meta 信息中补全轨道。
- 增加语言探测。
- 增加分辨率探测。
- 增加码率探测。
- 将信息统一映射到顶部条轨道菜单。
- 对 MPD 多 AdaptationSet、多音轨、多字幕场景进行专项测试。

### 第五阶段：诊断与日志增强

目标：让播放失败原因更容易判断，减少盲调。

建议增加：

- 播放失败类型分层：
  - libmpv 初始化失败；
  - HTTP 不可用；
  - 探测超时；
  - 找到源但播放失败；
  - 403/404/500；
  - CENC 解密失败；
  - EME 依赖；
  - 轨道超时；
  - 音频迟迟未就绪。
- 失败弹窗展示更具体的原因。
- 可配置日志级别。
- 一键导出本次播放诊断信息。

## 四、建议验收频道集

后续每轮改动后，建议用固定频道集回归，避免播放核心前进后又出现退化。

### 1. 普通 HLS

- 老版本 libmpv 可以播放的普通 m3u8/hls 频道。
- 验收点：
  - 有声音；
  - 有画面；
  - 切台无残留；
  - 停止播放后无后台声音。

### 2. 普通 MPD

- 不带 CENC 或不需要 Keys 的 MPD/DASH。
- 验收点：
  - 能起播；
  - 加载提示正常；
  - 播放失败时错误明确。

### 3. MPD/CENC + Keys

- 例如已经验证过的 `Channels\US.cfg` 中相关频道。
- 验收点：
  - 能起播；
  - 音视频最终同步；
  - 音频等待期间提示合理；
  - 超时策略不过早误判失败。

### 4. 混合协议频道文件

- 例如 `Channels\tennischannels.cfg`。
- 验收点：
  - 同一文件内 HLS、MPD 混合播放；
  - 不按文件格式误判协议；
  - 切换频道无串台。

### 5. 网页型 HLS

- 例如 `https://m.gdtv.cn/tvChannelDetail/43` / `https://www.gdtv.cn/tvChannelDetail/43`。
- 验收点：
  - 能嗅探真实直播流；
  - 能交给 libmpv 播放；
  - 探测超时可配置；
  - 探测失败可重试或浏览器兜底。

### 6. 明确失效链接

- 返回 `400`、`404`、`410`、`500`、`502`、`503`、`504` 的频道。
- 验收点：
  - 不长时间黑屏；
  - 提示“直播链接不可用/已不存在/服务器异常”；
  - 可选择播放下一个频道。

### 7. 浏览器 EME 依赖源

- Widevine/PlayReady 且没有 Keys 的频道。
- 验收点：
  - libmpv 先尝试；
  - 失败后提示浏览器播放；
  - 用户可选择默认浏览器、选择浏览器或取消。

## 五、优先级建议

短期优先级：

1. 播放切换稳定性回归。
2. 媒体解析器结构整理。
3. GDTv 类网页源成功率继续提升。
4. 轨道、语言、分辨率、码率探测。
5. iptv345 类复杂网站专项攻关。

当前建议先进入“稳定性回归 + 媒体解析器整理”阶段。这样可以先稳住主链路，再继续挑战更复杂的网页直播源。

# 2026-06-23 项目全面检视与阶段性总结

更新时间：2026-06-23

## 一、当前总体阶段判断

项目已经从“播放核心迁移和主链路打通”推进到“功能闭环基本成型、复杂源播放与工程质量持续强化”的阶段。

当前代码已经具备一个桌面播放器产品的主要骨架：

- PySide6 + libmpv 内置播放核心；
- 频道资源、本地资源、收藏、播放列表、浏览器播放、受控浏览器嗅探、诊断日志等主能力；
- Apple TV 风格玻璃拟态控制栏和统一化界面风格；
- GitHub 发布资料、宣传网页、图标、License、Windows 安装包工作流；
- 针对复杂源的解析、失败分类、请求头策略和日志体系已进入持续迭代状态。

下一阶段不宜继续无节制堆功能，建议转入“稳定性、诊断闭环、复杂源专项、自动化测试和发布质量”并行推进。

## 二、已完成的主要功能

### 1. 播放核心与渲染模式

已完成：

- 集成新版 `libmpv` 作为内置播放核心。
- 支持 HLS/M3U8、DASH/MPD、FLV、MP4、本地视频、本地音频、GIF、图片等资源类型。
- 支持本地渲染模式：流畅优先、画质优先、极致画质。
- 支持直播渲染模式：流畅优先、画质优先。
- 支持 MP4/FLV 点播回放策略 `http-file-vod`。
- 支持 `http-seekable=yes`，适配支持 Range 的回放视频。
- 支持播放、暂停、停止、进度拖动、音量、倍率、全屏、上一项/下一项。
- 修复过多轮播放卡顿、全屏双击误触暂停、停止失效、画面卡死等问题。

当前状态：

- 常规本地视频播放已经可用。
- 常规 HLS/MPD/FLV/MP4 播放链路已经可用。
- 复杂签名源、DRM 和网页动态源仍需专项提升。

### 2. 频道资源与本地资源管理

已完成：

- 支持打开资源目录。
- 支持加载频道资源文件：`m3u`、`m3u8`、`txt`、`json`、`cfg`。
- 支持本地媒体资源：视频、音频、GIF、图片。
- 资源库面板支持筛选、搜索、收藏按钮、长文件名省略和取消横向滚动条。
- 加载大目录已经从阻塞 UI 调整为后台扫描，明显降低界面冻结风险。

当前状态：

- 资源库作为“频道文件 + 本地媒体入口”的核心面板已经成立。
- 仍需继续优化超大目录扫描、取消扫描、增量刷新和错误反馈。

### 3. 频道列表与边缘滑入规则

已完成：

- 频道资源加载后形成频道分组和频道列表。
- 鼠标移动到左侧时，根据播放上下文决定滑入频道列表或资源库。
- 直播状态且存在已解析频道列表时滑入频道列表。
- 本地播放或无频道列表时滑入资源库。
- 从频道收藏进入直播但频道列表为空时滑入资源库。
- 即使频道播放失败，只要频道文件已经解析出频道列表，左侧仍滑入频道列表。
- 鼠标移动到右侧滑入本地播放列表面板。

当前状态：

- 左侧滑入规则已经比较符合实际使用逻辑。
- 后续可加入可视提示，例如“当前来源：频道列表/资源库/频道收藏”。

### 4. 智能解析与受控浏览器嗅探

已完成：

- `backend/stream_resolver.py` 提供统一解析入口。
- 支持直接媒体地址识别、HTTP 状态分类、302/301 跳转跟随、HTML/JSON/脚本候选提取、页面型源标记、解析缓存和解析取消。
- 受控浏览器嗅探支持 hook video src、fetch/XHR、Hls.js、mpegts、video.js，并捕获真实媒体 URL。
- 针对小红书回放 MP4 完成专项增强：
  - 浏览器式 `GET + Range` 探测；
  - Chromium 请求头伪装；
  - `sns-video*.xhscdn.com` 统一 HTTPS；
  - `redirect -> MP4` 清理 `Referer/Origin`，模拟浏览器地址栏直开 MP4。

当前状态：

- 普通媒体源和部分网页型源可自动解析。
- GDTV、小红书回放等场景已经取得进展。
- iptv345 等强会话/强上下文源仍是遗留专项。

### 5. 浏览器播放与浏览器选择

已完成：

- 工具栏恢复“默认浏览器观看”和“选择浏览器观看”按钮。
- 浏览器播放页面模板已适配打包环境。
- `live.html` 运行时写入用户数据目录，避免安装目录写权限问题。
- 浏览器解析接口按需解析，不再对大频道文件做全量预解析。
- 浏览器播放支持 EPG 查询和当前频道解析。

当前状态：

- 浏览器播放已成为 libmpv 失败时的重要兜底。
- 仍需继续改善 HLS 在浏览器端的兼容性、CORS、代理转发和错误提示。

### 6. 收藏管理

已完成：

- 资源收藏管理器支持收藏频道资源、视频、音频、GIF、图片。
- 资源收藏支持查看、播放、删除收藏关联、有效性检测、类型标注、类型筛选和搜索。
- 频道收藏管理器支持收藏单个频道、查看、播放、删除收藏关联、保存频道快照、搜索和直播类型标注。
- 资源列表和频道列表支持星标收藏/取消收藏。

当前状态：

- 收藏功能已完成主要闭环。
- 后续可增加批量清理失效收藏、导入导出、收藏分组。

### 7. 播放列表与本地专辑

已完成：

- 右侧播放列表面板。
- 支持多个本地播放专辑。
- 支持通过文件夹创建专辑。
- 支持自动连播。
- 支持专辑级设置：跳过片头、片头时长、跳过片尾、片尾时长、是否自动播放下一项。
- 播放控制栏上一项/下一项根据来源选择切换范围：频道列表、频道收藏、资源目录、资源收藏、播放专辑。

当前状态：

- 播放列表已具备基本产品形态。
- 仍需增强排序、拖拽、批量管理、播放历史和断点续播。

### 8. UI 风格与交互

## 三、当前仍需完善和增强的功能

### 1. 复杂网页源成功率

代表问题：

- iptv345 等需要页面触发、线路选择、Cookie/Session、同一浏览器上下文的源。
- 部分源在浏览器可以播放，但 libmpv 请求 m3u8 或分片返回 403/404。

建议方向：

- 浏览器探测代理播放；
- 复用受控浏览器 Cookie/请求头；
- 本地代理转发 m3u8 和分片；
- session 失效后自动重新探测；
- 多候选源评分和回退。

### 2. libmpv 失败诊断闭环

当前问题：

- `idle-active` 仍偏笼统。
- 用户需要人工查看日志才能判断是 403、TLS、demuxer、请求头还是签名过期。

建议方向：

- 读取 `mpv-runtime.log` 最后一段关键错误；
- 在 `playback.failed` 里附加 mpv 原始错误摘要；
- 失败弹窗显示更具体原因；
- 增加“一键复制诊断信息”。

### 3. 请求头策略配置化

当前问题：

- 自动策略已覆盖一部分源，但复杂 CDN 的 Referer/Origin/Cookie 策略差异很大。

建议方向：

- 频道级请求头策略：无 Referer、使用原始页 Referer、使用 origin Referer、自定义 Referer、自定义 Cookie、复制解析阶段请求头、使用受控浏览器代理上下文。
- UI 中提供高级编辑入口。

### 4. 自动化测试不足

当前问题：

- 主要依靠人工验证和少量 `py_compile`/轻量调用。
- 播放链路复杂，容易修一个源影响另一个源。

建议方向：

- 为 `stream_resolver` 建单元测试。
- 为 `playlist_manager` 建 M3U/TXT/JSON/CFG 解析测试。
- 为收藏和播放列表持久化建测试。
- 建立固定回归源样本集。
- CI 至少执行 Python 编译、核心单测和资源路径测试。

### 5. UI 主题继续收敛

当前问题：

- 大部分界面已统一，但仍存在一些控件内部局部 QSS。
- 长期维护时容易重新出现风格漂移。

建议方向：

- 继续把 overlay/dialog/button/list/tab 样式收敛到 `ui/theme.py`。
- 新增界面必须使用统一主题辅助函数。
- 建立界面验收截图清单。

### 6. 播放列表产品化

建议增强：

- 专辑内拖拽排序；
- 手动调整播放顺序；
- 播放历史；
- 断点续播；
- 批量移除失效资源；
- 当前播放项高亮；
- 专辑导入导出。

### 7. 轨道与媒体信息

当前已有：

- 基础轨道探测和顶部条菜单。

仍需增强：

- 更稳定地从频道文件读取视频/音频/字幕轨；

- MPD 多音轨、多字幕、多 AdaptationSet 专项测试。

### 8. 发布质量

建议增强：

## 四、下一阶段开发计划建议

### 阶段 A：稳定性与回归测试基线（优先级最高）

目标：降低“修复复杂源导致基础播放回退”的风险。

任务：

1. 建立 `tests/` 目录。
2. 增加解析器单元测试：
   - 直连 MP4；
   - 302 到 MP4；
   - HLS；
   - MPD；
   - HTML 内嵌媒体；
   - 404/403/500；
   - 拒绝 HEAD、支持 GET Range 的源。
3. 增加播放列表、收藏、资源类型识别测试。
4. GitHub Actions 增加测试步骤。
5. 固定一组人工验收频道和本地媒体样本。

建议周期：1 个小阶段。

### 阶段 B：复杂源诊断闭环

目标：让播放失败可以自解释。

任务：

1. 从 `mpv-runtime.log` 提取最近错误摘要。

建议周期：1 个小阶段。

### 阶段 C：浏览器探测代理播放

目标：攻克 iptv345 这类依赖浏览器会话的复杂源。

3. 代理转发 m3u8 和分片，并注入浏览器上下文。
4. 404/session 失效时自动重新探测。
5. 与普通 HLS/MPD/MP4 路径隔离，避免影响基础播放。

建议周期：2 个小阶段。

### 阶段 D：请求头策略和频道高级设置

目标：把复杂源人工修正能力产品化。

## 五、推荐近期执行顺序

建议下一步优先顺序：

1. 建立测试基线，先覆盖解析器和资源解析。
2. 做 mpv 失败诊断闭环，让后续复杂源定位更快。
3. 做浏览器探测代理播放，专项攻克 iptv345 等复杂网页源。
4. 做请求头策略配置化，提升复杂 CDN 源可维护性。
5. 再推进播放列表、本地媒体和 UI 细节增强。

这个顺序的好处是：先把“看清问题”和“不回退”做扎实，再继续扩大复杂源覆盖率。
