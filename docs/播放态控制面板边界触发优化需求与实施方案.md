# 播放态控制面板边界触发优化需求与实施方案

## 背景

当前在“面板滑入操作模式 = 鼠标移动触发”时，只要播放器处于播放状态，鼠标在播放区域内发生移动，就会唤起播放器顶部条和底部播放控制面板。这个逻辑虽然方便，但在观影过程中容易因为轻微移动鼠标而打断沉浸感，尤其是全屏观看、直播观看、长视频观看场景。

本次优化目标是：播放态下不再因为播放画面区域内任意鼠标移动而唤起顶部条和底部控制面板，只有鼠标进入顶部条或底部控制条对应的边界触发范围时才唤起。

## 需求理解

1. 仅影响“鼠标移动触发”模式。
2. “鼠标点击触发”模式保持当前逻辑，不受本次改动影响。
3. 仅调整播放状态下顶部条和底部播放控制面板的唤起方式。
4. 左侧频道/资源面板、右侧播放列表面板、顶部工具栏等边缘 hover 入口不在本次范围内，保持原有行为。
5. 控制面板已经显示时，鼠标在顶部条或底部控制条上移动，应保持显示并继续刷新自动隐藏计时，避免用户正在操作时面板消失。
6. 控制面板隐藏时，鼠标只有进入顶部或底部边界触发范围才唤起，不应在画面中央移动时唤起。

## 当前代码分析

相关代码集中在 `ui/player_panel.py`：

1. `mouseMoveEvent()` 当前在 hover 模式、播放态且未被侧边面板 suppress 时，直接调用 `_show_controls()`。
2. `_poll_cursor()` 当前只要检测到鼠标位于播放区域内，并且发生移动或处于已显示控制条范围内，就调用 `_show_controls()`。
3. `_position_controls()` 已经计算顶部条和底部控制条的真实 geometry，可复用这些 geometry 判断鼠标是否进入触发区域。
4. `_show_controls()` 会同时显示顶部条和底部条，并启动 3 秒自动隐藏计时。

核心问题是触发条件过宽：`inside and moved` 会把整个播放区域都视为控制条唤起区。

## 实施方案

### 1. 新增边界触发判定方法

在 `PlayerPanel` 中新增私有方法，例如：

```python
def _cursor_in_control_trigger_zone(self, local_pos):
    ...
```

建议判定规则：

1. 先调用 `_position_controls()`，确保顶部条和底部条 geometry 最新。
2. 顶部触发区：从播放区域顶部开始，向下扩展到 `max(top_bar.height(), 44)`，可额外增加 6-10px 容错。
3. 底部触发区：从播放区域底部向上扩展到 `max(bottom_bar.height() + bottom_margin, 96)`，兼容 120px 控制栏及底部留白。
4. 如果控制条已经可见，鼠标位于 `top_bar.geometry()` 或 `bottom_bar.geometry()` 内，也应返回 true。
5. 鼠标在画面中间区域返回 false。

### 2. 修改 `mouseMoveEvent()`

当前逻辑：hover 模式播放态任意鼠标移动就 `_show_controls()`。

修改为：

1. 仍只在 hover 模式、播放态、未 suppress 时生效。
2. 将事件坐标转换到 `video_stack_host` 局部坐标。
3. 只有 `_cursor_in_control_trigger_zone(local_pos)` 返回 true 时才调用 `_show_controls()`。
4. 如果不在触发区，不主动显示控制条。

### 3. 修改 `_poll_cursor()`

当前逻辑：`inside and (moved or over_bar)` 即显示。

修改为：

1. 继续保留 `click` 模式直接返回。
2. 继续保留播放态、suppress、mpv_widget 等保护条件。
3. 只有在播放区域内且鼠标进入顶部/底部触发区时，才调用 `_show_controls()`。
4. 控制条已显示时，鼠标位于控制条本体范围内，继续保持显示并刷新隐藏计时。
5. 鼠标在播放画面中央移动，不唤起控制条。

### 4. 不修改的内容

1. 不修改点击模式的边缘箭头逻辑。
2. 不修改左侧频道/资源面板和右侧播放列表的 hover 触发逻辑。
3. 不修改控制条样式、信号槽、播放功能。
4. 不修改快捷键逻辑，例如空格暂停/播放、回车全屏切换。

## 体验预期

1. 播放中鼠标在画面中央移动，不再弹出顶部条和底部控制条。
2. 鼠标移到顶部边缘附近，顶部条和底部控制面板一起显示。
3. 鼠标移到底部控制条区域附近，顶部条和底部控制面板一起显示。
4. 用户在控制条上操作时，控制条不会立刻消失。
5. 控制条显示后仍按原逻辑自动隐藏。

## 风险与边界

1. 如果触发区太窄，用户可能觉得控制条不容易唤起。
   - 解决：顶部触发区建议不少于 44px，底部触发区建议不少于 96px。
2. 如果底部控制条高度后续再次调整，触发区需要跟随 `bottom_bar.height()` 动态计算。
   - 解决：不要写死单一高度，使用 geometry 和最小阈值组合。
3. 全屏模式和窗口模式下坐标系必须一致。
   - 解决：统一使用 `video_stack_host.mapFromGlobal()` 或事件坐标映射到 host。
4. 侧边面板显示时，播放控制条仍应被 suppress。
   - 解决：保留现有 `_controls_suppressed` 逻辑。

## 验收方法

### 场景 1：hover 模式播放中画面中央移动

1. 设置“面板滑入操作模式”为“鼠标移动触发”。
2. 播放直播或本地视频。
3. 等待顶部条和底部控制条自动隐藏。
4. 在画面中央移动鼠标。
5. 预期：顶部条和底部控制条不出现。

### 场景 2：hover 模式顶部边界触发

1. 播放状态下等待控制条隐藏。
2. 将鼠标移动到播放画面顶部边缘附近。
3. 预期：顶部条和底部控制条显示。

### 场景 3：hover 模式底部边界触发

1. 播放状态下等待控制条隐藏。
2. 将鼠标移动到播放画面底部边缘或底部控制条预期区域附近。
3. 预期：顶部条和底部控制条显示。

### 场景 4：控制条操作保持显示

1. 唤起控制条后，将鼠标移动到播放按钮、进度条、音量或倍速控件上。
2. 预期：控制条保持显示，操作过程中不突然消失。

### 场景 5：点击模式不受影响

1. 设置“面板滑入操作模式”为“鼠标点击触发”。
2. 播放视频或直播。
3. 验证四个边缘箭头、空闲隐藏、鼠标移动恢复显示仍按现有逻辑工作。
4. 预期：本次 hover 优化不影响点击模式。

### 场景 6：侧边面板 suppress

1. 播放状态下滑入资源库、频道列表或播放列表。
2. 在播放区域顶部/底部移动鼠标。
3. 预期：侧边面板显示期间，播放控制条不抢焦点、不反复滑入。

## 建议实施顺序

1. 在 `PlayerPanel` 中新增触发区计算方法。
2. 修改 `mouseMoveEvent()` 使用触发区判定。
3. 修改 `_poll_cursor()` 使用触发区判定。
4. 运行 `python -m py_compile ui/player_panel.py ui/main_window.py`。
5. 按验收方法进行窗口模式和全屏模式手工验证。
