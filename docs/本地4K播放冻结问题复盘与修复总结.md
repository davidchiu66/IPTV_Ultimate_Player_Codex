# 本地 4K 播放冻结问题复盘与修复总结

## 背景

在本地播放 4K HEVC/10bit 视频时，播放器曾出现如下现象：

1. 窗口模式可播放数分钟后画面冻结。
2. 全屏模式更容易复现，通常 2 至 5 分钟后画面停止，声音和进度继续。
3. 暂停/继续无法恢复画面。
4. 点击停止后有概率导致界面卡死。
5. PotPlayer/VLC 在同一机器上播放正常。

## 排查过程

为定位问题，增加了两类诊断：

1. 应用诊断日志：记录播放策略、全屏状态、窗口/屏幕参数、播放快照。
2. 独立 libmpv 日志：`mpv_runtime.log`，用于观察 mpv 底层输出、解码、硬件加速状态。

关键日志显示：

```text
Failed to create SRV: GPU device instance has been suspended (0x887a0005)
Couldn't resize swapchain: GPU device instance has been suspended (0x887a0005)
AVHWFramesContext: Unable to lock D3D11VA surface (887a0005)
Could not copy back hardware decoded frame.
Error while decoding frame (hardware decoding)!
```

结论：问题根因不是全屏 UI、悬浮时钟或播放控制面板，而是本机 GPU/驱动在 D3D11/D3D11VA 硬解链路上不稳定。D3D11 原生输出和 D3D11VA copyback 都可能在持续播放后失败。

## 修复方案

本次将本地媒体播放策略改为稳定优先：

1. 本地媒体默认使用 `OpenGL + software decode`。
2. 本地播放 `smooth / quality / extreme` 三种模式均禁用硬解：`hwdec=no`。
3. 本地全屏不再触发渲染器切换和播放重载。
4. 画面停滞自动恢复时，切换到软件解码兼容模式，而不是 D3D11VA copyback。
5. 保留 D3D11VA copyback、D3D11 native 等配置作为后续手动/兜底研究对象，但不再作为本地播放默认路径。
6. 恢复悬浮时钟显示，经窗口和全屏播放验证，不再触发冻结。

## 直播 4K 是否需要同步修改

暂不建议一刀切修改直播 4K。

原因：

1. 当前已复现并确认的问题集中在本地 4K HEVC/10bit 文件，直播链路尚未复现相同冻结。
2. 直播 4K 的编码、码率、缓存、传输形态与本地文件不同，直接全局改为软件解码可能导致 CPU 压力过高、掉帧或直播延迟增加。
3. 直播当前仍走独立的 HLS/MPD/realtime 策略，未被本地播放软件解码策略影响。

后续建议：

1. 保持直播默认策略不变，继续使用 `auto-safe` 和直播专用缓存策略。
2. 如果直播 4K 也出现同类冻结，再增加“直播兼容模式/直播软件解码模式”设置项，而不是直接修改所有直播频道。
3. 在错误提示中增加 `887a0005` / D3D11VA 相关识别，提示用户切换到兼容播放模式。

## 日志收尾

本次排查期间临时开启了较多强制诊断日志。收尾后：

1. 全屏状态日志不再强制写入。
2. 本地播放快照不再强制写入。
3. 停滞检测和停止请求日志不再强制写入。
4. `mpv_runtime.log` 仅在设置中启用诊断日志且日志级别为“调试”时生成。

## 验收结果

已完成用户侧验证：

1. 窗口模式播放 4K 视频超过 10 分钟，没有冻结。
2. 全屏模式播放 4K 视频超过 10 分钟，没有冻结。
3. 恢复悬浮时钟后，全屏播放超过 10 分钟，没有冻结。

## 后续计划

1. 增加本地播放模式说明：稳定优先、质量优先、兼容模式的差异。
2. 后续可在设置中加入“硬件解码实验选项”，默认关闭。
3. 如需兼顾性能，可基于机器能力和实际日志做自动降级：D3D11VA 失败后持久化标记该设备使用软件解码。
4. 继续保留独立 mpv 日志能力，但默认关闭，避免普通用户运行时日志过多。
