# 播放器技术栈迁移与选型评估报告

## 背景

当前项目基于 `PySide6 + libmpv`，已经完成了直播、本地视频、MPD/CENC、HLS 嗅探、资源库、代理、轨道信息、播放控制栏等大量功能。

随着项目继续向商业级播放器方向演进，需要评估是否继续保持当前技术栈，或迁移到更高性能、更易打包、更利于长期维护的技术方案。

本文归档两份评估：

1. 从 `PySide6 + libmpv` 转向 `Qt C++ + libmpv` 的可行性评估。
2. 为提升性能、打包效率、开发效率，其他可选技术栈的横向评估。

---

## 一、Qt C++ + libmpv 迁移评估

### 结论

从 `PySide6 + libmpv` 转向 `Qt C++ + libmpv` 是可行的。

如果目标是长期维护、商业级播放器、复杂播放场景、本地 4K 和直播体验持续打磨，`Qt C++ + libmpv` 是一条更强的底层路线。

但这不是简单迁移，而是接近重构级别的技术栈切换。建议不要立即全量推倒重写，而是先做 Qt C++ 播放内核原型，验证关键能力后再决定是否正式迁移。

### 主要收益

1. **libmpv 集成更自然**

   libmpv 本身是 C API，C++ 调用更直接。生命周期、事件回调、日志、属性监听、轨道切换、播放状态同步都会比 Python 绑定更清晰。

2. **稳定性更好**

   当前 PySide6 + libmpv + 原生窗口叠加控件，容易遇到 z-order、窗口句柄、`QWidgetWindow must be a top level window`、`QPainter` 等问题。C++ 下可以更精细地控制窗口、线程和渲染路径。

3. **性能更好**

   C++ 没有 Python GIL、ctypes/绑定层开销。播放状态轮询、UI 动画、资源扫描、轨道信息解析等高频逻辑更适合长期运行。

4. **打包部署更专业**

   Windows 下 C++ Qt 程序配合 libmpv、ffmpeg、配置文件、资源文件，可以做成更标准的桌面播放器发布形态。启动速度、依赖体积和运行时环境可控性通常优于 Python 方案。

5. **UI 层长期可维护性更强**

   如果继续走商业级播放器路线，Qt C++ 的自绘控件、动画、OpenGL 渲染、快捷键、全屏、多显示器、系统托盘等能力更稳。

6. **有机会彻底解决部分叠层问题**

   如果使用 `mpv render API + QOpenGLWidget/QOpenGLWindow`，视频画面不再是外部原生子窗口，而是 Qt 渲染体系的一部分，顶部条、底部控制栏、侧边资源面板的层级问题会更容易处理。

### 主要风险

1. **迁移成本高**

   当前项目已有大量功能：频道文件解析、m3u/cfg/txt 混合协议、HLS 嗅探、浏览器探测、代理配置、轨道信息、资源库、本地视频、诊断日志、i18n、控制栏 UI。这些都要重新实现或迁移。

2. **不会自动解决 DRM/EME 问题**

   Qt C++ + libmpv 不会让 Widevine/PlayReady EME 频道天然可播。真正依赖浏览器 EME 的频道，仍然需要浏览器路径。

3. **开发复杂度上升**

   C++ 的内存管理、线程安全、异步任务、对象生命周期、崩溃定位都比 Python 更硬核。开发速度会比 PySide6 慢，UI 迭代成本更高。

4. **mpv 渲染方案要慎选**

   如果继续用 `wid` 嵌入 mpv 原生窗口，仍可能遇到覆盖层和窗口层级问题。若改用 `mpv render API`，架构更优，但需要处理 OpenGL、硬解、高 DPI、多屏、4K 性能等问题。

5. **浏览器嗅探迁移复杂**

   当前 Python 侧浏览器探测、DOM/Network 嗅探、超时重试、失败分类等逻辑迁到 C++ 会比较麻烦。Qt WebEngine C++ 可以做，但调试和维护成本不低。

6. **Qt 授权与部署需要评估**

   Qt LGPL/商业授权、动态链接方式、Qt WebEngine 分发、VC++ Runtime、libmpv/ffmpeg 许可证，都需要在发布前梳理清楚。

### 推荐迁移路线

1. **第一阶段：C++ 播放内核原型**

   验证：

   - libmpv 初始化；
   - HLS 播放；
   - MPD/CENC 播放；
   - 本地 4K 播放；
   - 播放、暂停、停止、seek、音量、倍速；
   - 轨道读取与切换。

2. **第二阶段：验证渲染方案**

   同时测试两种方案：

   - `wid` 嵌入：实现快，但层级问题可能继续存在；
   - `mpv render API`：架构更优，适合商业级播放器，但实现更复杂。

3. **第三阶段：迁移核心业务**

   优先迁移：

   - 播放状态机；
   - 资源文件解析；
   - 代理配置；
   - 轨道信息；
   - 日志诊断；
   - 本地视频播放策略。

4. **第四阶段：迁移复杂嗅探和浏览器路径**

   HLS 页面嗅探、GDTv/iptv345 等特殊源，建议最后迁移，因为这部分变化最大、风险最高。

5. **第五阶段：UI 重建**

   控制栏、资源面板、设置面板、频道详情等用 Qt C++ QWidget 重新实现。当前 PySide6 UI 可以作为视觉和交互原型参考。

## 二、其他技术栈选型评估

### 综合推荐

如果目标是同时提升性能、打包效率、开发效率，值得评估的方案包括：

1. `C# / .NET 8+ + Avalonia 或 WPF/WinUI + libmpv`
2. `Qt C++ + libmpv render API`
3. `Rust + Tauri/egui + libmpv`
4. 继续 `PySide6`，但将播放内核和重计算模块下沉到 C++/Rust 扩展

### 横向对比

| 方案 | 性能 | 打包 | 开发效率 | UI能力 | libmpv集成 | 推荐度 |
|---|---:|---:|---:|---:|---:|---:|
| Qt C++ + libmpv | 很高 | 高 | 中低 | 很强 | 很强 | 高 |
| C#/.NET + Avalonia/WPF + libmpv | 高 | 高 | 高 | 强 | 中高 | 很高 |
| Rust + Tauri/egui + libmpv | 很高 | 很高 | 中 | 中高 | 中高 | 中 |
| PySide6 + C++/Rust 播放内核 | 中高 | 中 | 高 | 强 | 中高 | 高 |
| Electron + mpv | 中 | 低 | 高 | 很强 | 中 | 不优先 |
| Flutter Desktop + libmpv | 中高 | 中高 | 中 | 强 | 中低 | 谨慎 |
| Go/Wails + libmpv | 中高 | 高 | 中 | 中 | 中 | 谨慎 |

## 三、重点方案分析

### 方案 A：C#/.NET + Avalonia/WPF + libmpv

这是一个很现实的替代方案，尤其适合 Windows 桌面播放器。

#### 收益

- 开发效率明显高于 C++。
- 性能比 Python 稳，UI 响应、线程、异步任务、资源扫描更好控。
- 打包比 Python 简洁，.NET 8/9 可以自包含发布。
- Windows 桌面生态成熟，调试体验好。
- 可以参考 `mpv.net` 这类成熟播放器路线。
- C# 调用 libmpv C API 比 Python 更直接，也比 C++ 低心智负担。
- UI 可选 WPF、WinUI 3 或 Avalonia。

#### 风险

- libmpv 嵌入窗口和覆盖层仍然要认真处理。
- 如果用原生窗口嵌入，仍可能遇到层级问题。
- 要实现类似当前玻璃拟态控制栏，需要较多自定义控件。
- Qt/PySide 现有 UI 不能直接迁移，需要重写 UI 层。

#### 判断

如果希望兼顾开发效率和长期性能，`C#/.NET + Avalonia/WPF + libmpv` 是比 Qt C++ 更平衡的路线。

---

### 方案 B：Qt C++ + libmpv render API

这是性能和底层控制力最强的路线。

#### 收益

- libmpv 集成最自然。
- 可以使用 `mpv_render_context`，把视频画面融入 Qt 渲染体系。
- 更容易彻底解决 mpv 原生窗口和 QWidget 覆盖层冲突。
- 本地 4K、直播、轨道、硬解、全屏、多屏控制都更有底层掌控力。

#### 风险

- 开发效率低于 C# 和 Python。
- 当前功能迁移成本最高。
- 浏览器嗅探、配置、资源库、i18n、诊断系统都要重写。

#### 判断

如果目标是最终商业级播放器内核，这个方向最好；如果目标是快速迭代功能，不如 C# 或混合方案。

---

### 方案 C：PySide6 + C++/Rust 内核模块

这是最稳的渐进方案。

#### 做法

- UI 继续 PySide6。
- 播放内核、libmpv 绑定、轨道探测、资源扫描、高性能解析模块，逐步用 C++/Rust 写成扩展。
- Python 继续负责业务编排、配置、界面和快速迭代。

#### 收益

- 不推倒重来。
- 当前功能保留最大。
- 性能瓶颈可以逐个拆。
- 风险最小。

#### 风险

- 打包仍然比原生方案复杂。
- Python 主程序的启动速度和依赖体积仍在。
- UI 层和 mpv 窗口层级问题不一定彻底消失。

#### 判断

短中期最务实。可以先把 libmpv 控制封装成 C++/Rust 模块，再决定是否全量迁移。

---

## 四、不优先推荐的方案

### Electron + mpv

开发效率高，但对当前项目不一定划算。

主要问题：

- 包体大。
- 内存占用高。
- libmpv 嵌入和原生窗口层级仍复杂。
- 当前已经有受控浏览器探测逻辑，播放器主界面再上 Electron，架构会变重。
- 对本地 4K 播放器来说，Electron 不是天然优势。

除非未来明确要把 UI 完全 Web 化，否则不优先。

### Flutter Desktop + libmpv

UI 能力强，但 libmpv 原生集成和复杂播放器控制链路不如 Qt/C# 直接。

### Go/Wails + libmpv

打包体验不错，但复杂桌面播放器 UI、原生视频窗口、轨道控制和浏览器探测生态不如 Qt/C# 成熟。

---

## 五、推荐决策路径

### 短期

继续保持 `PySide6 + libmpv`，完成当前播放器功能打磨，同时记录架构痛点。

### 中期

做两个小原型对比：

1. `C# + libmpv`

   验证 HLS、MPD/CENC、本地 4K、轨道读取、播放控制、打包体积。

2. `Qt C++ + mpv render API`

   验证 HLS、MPD/CENC、本地 4K、覆盖层、播放控制栏、资源面板叠层。

### 长期

根据原型结果选择：

- 如果更重视商业级内核和底层控制：选择 `Qt C++ + libmpv render API`。
- 如果更重视开发效率、Windows 桌面体验和打包效率：选择 `C#/.NET + Avalonia/WPF + libmpv`。
- 如果希望最小风险持续演进：选择 `PySide6 + C++/Rust 内核模块`。

---

## 六、最终建议

不建议现在立即推倒重写。

建议先规划技术验证分支，重点验证：

- libmpv 初始化与生命周期；
- HLS/MPD/CENC 播放；
- 本地 4K 播放性能；
- 轨道读取与切换；
- 硬解策略；
- 全屏与多屏；
- 覆盖层与控制栏叠层；
- 打包体积与启动速度；
- 浏览器嗅探路径迁移成本。

如果两个原型都跑通，再根据实际开发体验、画面稳定性、覆盖层实现难度、打包体积和性能数据，决定主路线。
