# 遗留问题清单

## 1. iptv345.com 页面嗅探后 libmpv 播放仍返回 404

### 当前状态

暂缓处理，后续作为专项问题继续分析。

### 现象

- 频道示例：`https://iptv345.com/?act=play&token=...&tid=ys&id=1`
- 网页内可以播放。
- 程序受控浏览器探测已经能够触发页面播放逻辑，并捕获真实 HLS 地址。
- libmpv 播放捕获到的 `index.m3u8?session=...` 时仍报：
  - 协议：`HLS/M3U8`
  - 代码：`source-dead`
  - HTTP：`404`

### 已确认信息

- 页面需要先触发播放逻辑，否则不会生成最终 m3u8。
- `play.php?...` 是入口/鉴权地址，不应直接作为最终播放地址。
- 受控浏览器已能捕获类似地址：
  - `https://t1.iptv200.com/live/cctv1/index.m3u8?session=...`
  - `https://t2.iptv200.com/live/cctv1/index.m3u8?session=...`
- IDM 在网页播放后也能嗅探到同类 `index.m3u8?session=...` 地址。
- 程序已尝试补充浏览器 UA、`Referer`、`Origin`、`Accept` 等请求头，但 libmpv 仍返回 404。

### 已完成尝试

- 增强受控浏览器探测，自动触发页面播放按钮。
- hook `Hls.loadSource()` 和 `mpegts.createPlayer()`。
- 从 XHR/fetch 的 `responseURL` 捕获最终 m3u8。
- 避免把 `.jpeg`、`.jpg` 等静态资源误判为直播源。
- 将 `play.php` 降级为入口 URL，不再作为最终可播放源。
- 播放前为 libmpv 补充浏览器风格请求头。

### 初步判断

该站点可能不仅校验 m3u8 URL 和请求头，还依赖以下上下文之一：

- 浏览器会话 Cookie。
- `play.php` 入口链路中的服务端会话状态。
- m3u8 地址和分片请求的短生命周期绑定。
- 请求来源、IP、连接顺序或同一会话内的连续请求。
- 需要复用受控浏览器探测阶段的网络上下文。

### 后续方案方向

优先考虑“浏览器探测代理播放”：

1. 受控浏览器完成页面触发和 m3u8 捕获。
2. 程序启动本地代理地址交给 libmpv。
3. libmpv 请求本地代理。
4. 本地代理按受控浏览器捕获到的入口链路、请求头、Cookie、Referer、Origin 等上下文去请求真实 m3u8 和分片。
5. 对 session 失效或 404 自动重新探测并刷新源。

### 验收标准

- iptv345.com 示例频道可在 libmpv 内播放。
- 切换频道后不会串台。
- session 失效时能重新探测或给出明确错误。
- 不影响普通 HLS、MPD、m3u8 文件频道播放。
