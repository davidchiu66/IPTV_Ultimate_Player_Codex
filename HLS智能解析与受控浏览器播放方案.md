## 5. 统一解析流程

### 5.1 第1层：快速探测

优先执行：

1. `HEAD`
2. 如果 `HEAD` 不可靠、不被支持或返回结果不完整，再执行有限字节 `GET`

采集信息：

1. HTTP 状态码
2. 重定向后的最终 URL
3. `Content-Type`
4. 响应体前若干 KB

### 5.1.1 状态码总体处理原则

媒体探测不能只把 `200 OK` 视为正常。

对于直播流、页面型播放页、CDN 回源和跳转链路，以下状态码都可能属于正常或符合预期的响应结果：

#### 成功类（2xx）

- `200 OK`
- `201 Created`
- `202 Accepted`
- `203 Non-Authoritative Information`
- `204 No Content`
- `205 Reset Content`
- `206 Partial Content`

#### 重定向类（3xx）

- `301 Moved Permanently`
- `302 Found`
- `303 See Other`
- `304 Not Modified`
- `307 Temporary Redirect`
- `308 Permanent Redirect`

因此统一解析器不能写成“如果是 200 才继续”，而应采用：

1. `2xx`：按具体语义继续判断
2. `3xx`：优先跟随或记录重定向后再判断
3. `4xx`：区分失效、拒绝、鉴权、限流等类型
4. `5xx`：视为源站或网关异常，走短期失败缓存和用户提示

### 5.1.2 各类状态码处理建议

#### A. 2xx 处理建议

##### `200 OK`

正常进入后续识别流程：

1. 判断是否为直接媒体
2. 若是 HTML，再进入静态页面嗅探

##### `201 / 202 / 203`

虽然直播场景不常见，但仍应视为“请求成功或已被接受”：

1. 继续读取 `Content-Type`
2. 继续识别响应体特征
3. 不要直接当作失败

##### `204 No Content`

处理建议：

1. 不应直接认定为媒体可播放
2. 对直播探测而言，应标记为“请求成功但无可用内容”
3. 返回 `unresolved` 或 `error`
4. 给出提示：源存在响应，但当前无媒体内容

##### `205 Reset Content`

处理建议：

1. 与 `204` 类似，不视为直接可播放
2. 可标记为“无可用媒体内容”

##### `206 Partial Content`

这是直播流和 CDN 场景下非常重要的正常状态。

处理建议：

1. 视为有效响应
2. 继续做媒体类型识别
3. 如果响应体片段符合 `m3u8/mp4/flv/ts` 特征，应直接当作有效媒体流

#### B. 3xx 处理建议

##### `301 / 302 / 303 / 307 / 308`

处理建议：

1. 默认视为“目标存在且要求跳转”
2. 解析器应尽量跟随跳转
3. 记录最终 URL
4. 对最终 URL 再继续媒体识别或 HTML 嗅探

若跳转次数过多：

1. 视为异常链路
2. 返回 `error`
3. 提示“链接重定向异常”

##### `304 Not Modified`

这是缓存命中场景的正常结果。

处理建议：

1. 不应判死
2. 如果本地已有缓存解析结果，应优先复用缓存
3. 若无缓存但收到 `304`，可降级为重新发起无缓存探测请求

#### C. 4xx 处理建议

##### 明确失效类

- `404 Not Found`
- `410 Gone`
- `451 Unavailable For Legal Reasons`

处理建议：

1. 直接判定为 `dead`
2. 不进入播放器
3. 提示“直播链接已不存在或不可用”

##### 鉴权/拒绝类

- `401 Unauthorized`
- `403 Forbidden`

处理建议：

1. 不直接归类为链接失效
2. 应提示“源存在，但当前请求未授权或被拒绝”
3. 保留后续通过 Header、Cookie、Referer、Token 修正的可能

##### 限流/请求条件类

- `405 Method Not Allowed`
- `406 Not Acceptable`
- `408 Request Timeout`
- `409 Conflict`
- `412 Precondition Failed`
- `416 Range Not Satisfiable`
- `429 Too Many Requests`

处理建议：

1. 视为“探测失败但不等于源失效”
2. 根据状态码给出更细提示
3. 建议走短期失败缓存，避免频繁重试

其中：

- `405` 说明 `HEAD` 可能不被支持，此时应自动降级到 `GET`
- `416` 在 Range 探测时可能出现，不应直接判死，应尝试普通 `GET`
- `429` 说明被限流，应延长失败缓存时间并提示稍后再试

#### D. 5xx 处理建议

常见包括：

- `500`
- `502`
- `503`
- `504`

处理建议：

1. 视为源站、CDN 或网关异常
2. 不直接认定为永久失效
3. 应做短期失败缓存
4. 提示“源暂时异常，可稍后重试或切换其他频道”

### 5.2 第2层：直接媒体识别

若满足以下任一条件，直接输出真实媒体：

1. URL 或重定向地址包含 `.m3u8`
2. `Content-Type` 为 `application/vnd.apple.mpegurl` 或同类 HLS 类型
3. 响应文本以 `#EXTM3U` 开头
4. `Content-Type` 为 `video/mp4`
5. 文件头识别为 FLV

### 5.3 第3层：按状态码分流判定

统一解析器在完成快速探测后，应先按状态码做分流，而不是简单写成“非 200 即失败”。

建议分流如下：

1. `2xx`：继续媒体识别或页面嗅探
2. `3xx`：优先跟随跳转或复用缓存，再继续识别
3. 明确失效类 `4xx`：直接判定 `dead`
4. 鉴权/限流/请求条件类 `4xx`：判定为 `error` 或 `unresolved`，不直接归类为永久失效
5. `5xx`：判定为源异常，走短期失败缓存

其中可直接判定为 `dead` 的典型状态码：

- `404`
- `410`
- `451`

### 5.4 第4层：静态页面嗅探

若满足以下条件，则进行静态页面嗅探：

1. 状态码属于可继续处理的成功响应或正常跳转结果
   - 例如 `200/201/202/203/206`
   - 或已跟随完成的 `301/302/303/307/308`
   - 或可复用缓存结果的 `304`
2. 响应内容看起来是 HTML 页面，而不是直接媒体流

嗅探目标包括：

1. `.m3u8`
2. `.mp4`
3. `.flv`
4. 常见字段：
   - `source`
   - `file`
   - `url`
   - `playlist`
   - `manifest`
   - `stream`
   - `hls`
5. 常见播放器配置：
   - `jwplayer`
   - `clappr`
   - `videojs`
   - `shaka`
   - `hls.js`

### 5.5 第5层：深度 JS 嗅探

如果静态嗅探未命中，但页面明显属于播放器站点，则返回：

- `status = page`
- `need_js_probe = true`

后续由深度 JS 嗅探模块处理。

## 14. 验收方法

验收分为四组。

### 14.1 内置播放器基础验收

准备样例：

1. 明确 `.m3u8` 后缀频道
2. 无后缀但实际返回 HLS 的频道
3. 返回 HTML 且页面中可静态提取 `m3u8` 的频道
4. `206 Partial Content` 的有效直播源
5. 301/302 跳转后返回真实 HLS 的频道
6. 304 缓存命中场景
7. 404 失效频道
8. 401/403 鉴权失败频道
9. 429 限流频道
10. 500/502 不稳定频道

验收步骤：

1. 在主程序中逐个播放以上频道
2. 观察加载状态提示是否分阶段显示
3. 检查 `206` 源是否被正确识别为有效媒体流
4. 检查跳转源是否能跟随到最终媒体地址
5. 检查缓存命中场景是否不会误判失效
6. 检查成功频道是否最终交给 `libmpv` 播放
7. 检查失效频道是否直接提示，不进入黑屏长等待

通过标准：

1. 明确媒体源可以直接起播
2. 页面型静态可嗅探源可以自动找到真实流并起播
3. `206/3xx/304` 等有效状态不会被误判失败
4. `404/410/451` 与 `401/403/429/5xx` 会被区别处理

### 14.2 浏览器模式验收

准备样例：

1. 小频道文件（3~5个频道）
2. 大频道文件（200+频道）
3. 混合频道文件（HLS/MP4/FLV/无后缀混合）

验收步骤：

1. 打开浏览器播放模式
2. 验证浏览器初次打开速度
3. 在频道列表中随机切换多个频道
4. 观察是否每次只按需解析当前频道
5. 检查是否仍然存在网页容器播放行为

通过标准：

1. 打开浏览器时不出现全量预解析等待
2. 大频道文件不会因预解析导致明显卡顿
3. 浏览器端播放使用真实媒体地址，而非网页容器

### 14.3 失效链路验收

准备样例：

1. 404 频道
2. 410 频道
3. 451 频道
4. 401/403 频道
5. 405 或 416 频道
6. 429 频道
7. 500/502/503/504 频道

验收步骤：

1. 在内置播放器中播放
2. 在浏览器模式中播放
3. 观察提示信息和按钮
4. 检查不同状态码是否提示不同类型的问题
5. 检查“播放下一个频道”是否可用

通过标准：

1. 明确失效源能快速失败
2. 鉴权失败、限流、源异常能和“链接已不存在”区分提示
3. 不出现长时间无反馈黑屏

### 14.4 缓存与性能验收

验收步骤：

1. 同一频道连续播放两次
2. 短时间内多次切回同一频道
3. 大频道列表中切换不同频道并回切

通过标准：

1. 已解析频道再次播放明显更快
2. 失效频道不会短时间内被疯狂重复探测
3. 不会在打开频道文件时对全部频道发起探测请求
